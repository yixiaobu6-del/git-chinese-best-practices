# Git 常见坑合集

> 盘点 Git 使用中最容易踩的坑及解决方案

## 一、提交相关的坑

### 坑1：提交信息写错

**错误现场**：
```bash
git commit -m "这是一个错自"
# 提交后才看到拼写错误
```

**解决方案**：
```bash
# 方法一：修改最近一次提交信息
git commit --amend -m "这是一个正确的提交信息"

# 方法二：进入编辑器修改
git commit --amend
```

### 坑2：提交后发现漏了文件

**错误现场**：
```bash
git commit -m "完成用户登录功能"
# 提交完了才发现还有一个文件没加进去
```

**解决方案**：
```bash
# 添加漏掉的文件并合并到上次提交
git add 漏掉的文件
git commit --amend --no-edit
```

**注意**：
- 如果已经推送到远程仓库，需要 `git push --force-with-lease`
- 强烈建议只在个人分支上使用 force push

### 坑3：把敏感信息提交了

**错误现场**：
```bash
# 不小心把密码/key提交到仓库
git add .
git commit -m "更新配置"
```

**解决方案**：
```bash
# 1. 删除文件并修改历史
git rm --cached config.env
echo "config.env" >> .gitignore
git commit -m "移除敏感文件"

# 2. 如果需要彻底删除历史中的记录
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch config.env" \
  --prune-empty --tag-name-filter cat -- --all
```

**重要**：如果已经推送到远程，立即更换密码和密钥！

## 二、分支相关的坑

### 坑4：在错误的分支上做了修改

**错误现场**：
```bash
# 直接在 master 分支上改了代码开始开发
# 改到一半发现应该在 feature 分支上
```

**解决方案**：
```bash
# 方法一：还没有提交
git stash
git checkout feature
git stash pop

# 方法二：已经提交了一次（但尚未推送）
git checkout -b feature
git checkout master
git reset HEAD~1 --hard
```

### 坑5：分支合并冲突

**错误现场**：
```bash
git merge feature
# CONFLICT: 文件有冲突
```

**解决方案**：
```bash
# 1. 查看冲突文件
git status

# 2. 手动解决冲突后
git add 已解决的文件
git commit -m "解决合并冲突"

# 3. 如果不小心搞乱了，可以回退
git merge --abort
```

### 坑6：删除了未合并的分支

**错误现场**：
```bash
git branch -D feature
# 删完才发现上面的改动还没合并
```

**解决方案**：
```bash
# 可以通过 reflog 找到分支最后的提交
git reflog
# 找到对应的 commit hash
git checkout -b feature <commit-hash>
```

## 三、撤销相关的坑

### 坑7：reset 和 revert 分不清

**reset（适用于未推送的提交）**：
```bash
# 撤销提交，保留修改
git reset --soft HEAD~1

# 撤销提交，丢弃修改
git reset --hard HEAD~1

# 撤销到指定提交
git reset --hard <commit-hash>
```

**revert（适用于已推送的提交）**：
```bash
# 创建一个新的提交来撤销指定的提交
git revert <commit-hash>

# 多人协作时，永远用 revert 而不是 reset
```

### 坑8：后悔了 hard reset

**错误现场**：
```bash
git reset --hard HEAD~1
# 发现刚才的修改全没了，后悔了...
```

**解决方案**：
```bash
# 通过 reflog 找回
git reflog
# 找到 reset 之前的 commit hash
git reset --hard <commit-hash>
```

## 四、远程相关的坑

### 坑9：push 被拒绝

**错误现场**：
```bash
git push origin main
# ! [rejected] main -> main (non-fast-forward)
```

**解决方案**：
```bash
# 先拉取远程修改
git pull origin main

# 如果有冲突，解决冲突后重新推送
git push origin main

# 如果确定要覆盖远程（慎用）
git push --force-with-lease origin main
```

### 坑10：pull 导致了不必要的合并提交

**错误现场**：
```bash
git pull origin main
# 生成了一个 "Merge branch 'main'" 的提交
```

**解决方案**：
```bash
# 使用 rebase 模式拉取
git pull --rebase origin main

# 设置为默认行为
git config --global pull.rebase true
```

## 五、暂存相关的坑

### 坑11：stash 太多记不清了

**错误现场**：
```bash
git stash
# 存了很多次后，分不清哪个是哪个
```

**解决方案**：
```bash
# 查看 stash 列表
git stash list

# 给 stash 加描述
git stash push -m "登录功能_完成80%"

# 指定编号应用
git stash apply stash@{2}
```

## 六、其他常见坑

### 坑12：大文件提交

**问题**：不小心提交了大的二进制文件，导致仓库体积膨胀

**解决方案**：
```bash
# 方案一：如果还没有推送
git rm --cached 大文件
echo "*.zip" >> .gitignore
git commit --amend

# 方案二：使用 Git LFS（推荐）
git lfs track "*.psd"
git add .gitattributes
```

### 坑13：换行符问题

**问题**：Windows 和 macOS/Linux 的换行符不一致

**解决方案**：
```bash
# macOS/Linux
git config --global core.autocrlf input

# Windows
git config --global core.autocrlf true

# 或者在项目根目录创建 .gitattributes
* text=auto
*.js text eol=lf
*.json text eol=lf
```

### 坑14：cherry-pick 后的冲突

**错误现场**：
```bash
git cherry-pick <commit-hash>
# 发生冲突
```

**解决方案**：
```bash
# 解决冲突后
git add .
git cherry-pick --continue

# 或者放弃 cherry-pick
git cherry-pick --abort
```

## 七、预防措施清单

### 提交前检查
- [ ] 确认没有敏感信息
- [ ] 确认没有大文件
- [ ] 确认提交信息正确
- [ ] 确认文件变更完整

### 推送前检查
- [ ] 确认分支名称正确
- [ ] 确认已经是最新代码
- [ ] 确认所有冲突已解决
- [ ] 确认 CI 测试通过

### 撤销前检查
- [ ] 确认修改是否已推送
- [ ] 确认他人是否基于此修改
- [ ] 确认是否有备份（reflog）

## 八、常用救命命令

```bash
# 找回一切
git reflog

# 取消暂存
git restore --staged 文件名

# 丢弃工作区修改
git restore 文件名

# 临时存储工作
git stash

# 应用最后一条 stash
git stash pop
```
